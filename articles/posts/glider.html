What does it take to build a reliable long range UAV control system?

Quite alot.
But theres a very well defined starting point.
A sufficient amount of processing power is required to handle all of the complex calculations the control system needs to maintain it's various feedback loops.
So suffice it to say, a powerful processing unit is crucial.

    ATmega328P (Uno) - 16Mhz
    ATmega2560 (Mega) - 16Mhz
    MK20DX256VLH7 (Teensy 3.1/3.2) - 72Mhz
      Overclocked: 96Mhz
    STM32F10C8T6 -72Mhz

Most of the entry level developer boards such as the Arduino Nano, Uno and Mega are considered underpowered in relation to an ideal clock speed range of 70-150Mhz

For that reason, the ESP8266 NodeMCU based on the ESP-12E module is a top-contender in this category.
Due to packing both WiFi capabilities and a very flexible MCU.

    Wireless Standard: IEEE 802.11 b/g/n
    Serial Transmission: 110 - 921600 baud
    Network Protocol: IPv4, TCP / UDP / FTP / HTTP
    Input Voltage: 7-12V
    Operating Voltage: 3.3V
    Operating Current: 80 mA
    Digital I/O Pins (DIO): 16
    Microcontroller: Tensilica 32-bit RISC CPU Xtensa LX106
    Flash Memory: 4 MB
    Memory: 32KiB instructions, 80KiB user-data RAM
    Factory Clock Speed: 80 MHz
      Overclocked: 160Mhz!!!

Lets talk sensors

In order to estimate the aircraft's attitude and position in space, the system uses a 9-axis DOF(degree-of-freedom) IMU as well a barometric pressure sensor.
    Inertial Measurement Unit: MPU-9250 (Gyroscope + Accelerometer + Magnetometer)
    Barometric Sensor: BMP-280 (Pressure + Altitude)
The entire control system handles all its sensor communication over I2C, for fast communication while using the least amount of GPIO pins.

Flight Dynamics

A technical overview of the aircraft's flight dynamics are discussed here:

Note: The NED(North-East-Down) fixed-body coordinate system is used throughout
The equations of motion for a aircraft can be developed using Newton's second law for each differential element of mass in the aircraft \[d \overrightarrow F = \overrightarrow a dm\] then integrating over the entire aircraft.
The time rate of change of the coordinates in the inertial frame instantaneously coincides with the body axes and are represented as: \[ \dot x = u + qz - ry \] \[ \dot y = v + rx - pz \] \[ \dot z = w + py - qx \] where \((u,v,w)\) are the linear velocities & \((p,q,r) \) are the rotation rates about the axis.
The corresponding accelerations are: \[\ddot x= \frac{d}{dt}(u+qz-ry) \] \[\ddot y= \frac{d}{dt}(v+rx-pz) \] \[\ddot z= \frac{d}{dt}(w+py-qx) \] or \[\ddot x= \dot u+\dot qz+q(w+py-qx)-\dot ry-r(v+rx-pz)\] \[\ddot y= \dot v+\dot rx+r(u+qz-ry)-\dot pz-p(w+py-qx)\] \[\ddot z= \dot w+\dot py+p(v+rx-pz)-\dot qx-q(u+qz-ry)\] Therefore, the net product of mass time acceleration on the entire aircraft is $$ \begin{align} m\overrightarrow a =& \int_{m} \{[ \dot u+\dot qz+q(w+py-qx)-\dot ry-r(v+rx-pz)]\hat i+ \\ & [\dot v+\dot rx+r(u+qz-ry)-\dot pz-p(w+py-qx)]\hat j+ \\ & [\dot w+\dot py+p(v+rx-pz)-\dot qx-q(u+qz-ry)]\hat k \}dm \end{align} $$ At this point the velocity and accelerations, are linear and angular, and are constant during the integration over the aircraft coordinates $$ \int_m dm = m $$ which integrates to the aircraft mass \(m\), $$\int_m xdm = \int_m y dm = \int_m zdm = 0 $$ they are all zero because the origin of the coordinate system is at the aircraft's center of mass \(x_{cm}\) $$m\overrightarrow a = m[(\dot u+qw-rv)\hat i+(\dot v+ru-pw)\hat j+(\dot w +pv-qu)\hat k]$$ To arrive at Newton's Second Law, we set \(m \overrightarrow a \) equal to the net external force acting on the aircraft. This force is the sum of the aerodynamic (incl. propulsive) forces and those due to gravity.
To characterize the orientation of the body axis system \(wrt.\) the gravity vector. Orientation is described using Euler's Angles of the body system \(wrt.\) the inertial system\((x_f ,y_f ,z_f)\)
  • \(z_f\) axis points down
  • \(x_f\) axis point North
  • \(y_f\) axis points East
    • The aircraft's orientation is determined by rotating through the heading angle \(\psi\), then pitching up through pitch angle \(\theta\), then rolling through \(\phi\).
    Since we want the orientation of the gravity vector, the first rotation is ignored. The second rotation(pitch) can be represented as $$ \begin{pmatrix}x_2 \\ y_2\\z_2\end{pmatrix} = \begin{pmatrix} \cos\theta & 0 & -\sin\theta\\ 0 & 1 & 0\\ \sin\theta & 0 & \cos\theta \end{pmatrix} \begin{pmatrix} x_f\\y_f\\z_f \end{pmatrix} $$ third rotation(roll), $$ \begin{pmatrix} x\\y\\z \end{pmatrix} = \begin{pmatrix} 1&0&0\\0&\cos\phi&\sin\phi\\0&-\sin\phi&\cos\phi \end{pmatrix} \begin{pmatrix} x_2\\y_2\\z_2 \end{pmatrix} $$
    Notation example:
    examine a vector \(x\) expressed in the body frame \(b\), denoted by \(x^b\). The rot. matrix \(R^{nb}\) rotates the vector from the body frame \(b\) to the navigation frame \(n\).
    The rot. from the navigation frame \(n\) to body frame \(b\) is expressed as \(R^{bn}=(R^{nb})^T\)
    \(x^n=R^{nb}x^b\), \(x^b=(R^{nb})^T x^n=R^{bn}x^n\).
    Gyros measure the angular velocity of the body frame \(wrt.\) the inertial frame, written as \(w_{ib}^b\). Angular velocity is expressed as, $$ w_{ib}^b = R^{bn}(\omega_{ie}^n + \omega_{en}^n)+\omega_{nb}^b, $$ where \(R^{bn}\) is the rot. matrix from the navigation frame to body frame.
    The accelerometer measures the specific force \(f\) in the body frame \(b\) $$ f^b = R^{bn}(a_{ii}^n - g^n), $$ where \(g\) is the gravity vector and \(a_{ii}^n\) is linear acceleration expressed in the navigation frame, expanded as $$ a_{ii}^n = R^{ne}R^{ei}a_{ii}^i $$ For navigation we are interested in the position of the sensor in the navigation frame \(p^n\), it's derivative are preformed in the navigation frame $$ \frac{d}{dt}x^u\mid_n = v_n^n,\;\;\;\; \frac{d}{dt}v^n\mid_n = a^n_{nn} $$

    Using the IMU to determine Position & Orientation

    To obtain the raw values from the IMU and Barometric sensor I used a websocket server to stream sensor data to a host PC for processing during the design phase before updating any microcontroller firmware.
    The portion of C++ code that handles this on the ESP8266 is shown below 

    #include <WebSocketsServer.h>

WebSocketsServer webSocket = WebSocketsServer(80);
char buffer_udp_in[150];
char buffer_udp_out[150];

// Called when receiving websocket packet
void onWebSocketEvent(u_int num, WStype_t type, uint8_t *payload, size_t length)
{
  // Determine type of Websocket event
  switch (type)
  {
  case (WStype_DISCONNECTED):
  {
    Serial.printf("[%u] Disconnected.\n", num);
    break;
  }
  case (WStype_CONNECTED):
  {
    IPAddress ip = webSocket.remoteIP(num);
    Serial.printf("[%u] Connection from ", num);
    Serial.println(ip.toString());
    break;
  }
  case (WStype_TEXT):
  {
    sprintf(buffer_udp_in, "%s", payload);
    Serial.println(buffer_udp_in);
    sprintf(buffer_udp_out, "%.3f,%.3f,%.3f,%.3f,%.3f,3f,%.3f,%.3f,%.3f,3f,%.3f,3f,%.3f,%.3f,%.3f,%.3f,%.3f,%.3f,%.3", acc[0], acc[1], acc[2], gyro[0], gyro[1], gyro[2], mag[0], mag[1], mag[2], qrt[0], qrt[1], qrt[2], qrt[3], pressure, altitude, temp, pitch, roll, yaw);
    webSocket.sendTXT(num, buffer_udp_out);
    break;
  }
  case (WStype_BIN):
  case (WStype_ERROR):
  case (WStype_FRAGMENT_TEXT_START):
  case (WStype_FRAGMENT):
  case (WStype_FRAGMENT_FIN):
  default:
    break;
  }
}
void start_websocket_loop()
{
  webSocket.loop();
}

    The host receives the data via a python websocket. Then the data can then be processed and visualized.
    The python websocket is as follows 
    import websocket

class TCP_Manager:
    def __init__(self):
        self.targetIP = "192.168.0.25"
        self.handler = websocket.WebSocket()
        try:
            self.handler.connect(f"ws://{self.targetIP}")
            print(f"Connected to IP: {self.targetIP}")
        except Exception as error:
            print(error)
    
    def send_data(self, data_type, data):
        '''Send data to tagertIP, return True if sent else Return False'''
        packet_out = (str(data_type) + ":" + str(data))
        try:
            self.handler.send(packet_out)
            return True
        except Exception as error:
            print(error)
        return False

    def receive_data(self):
        '''Receive packet from targetIP return the received packet as an array'''
        packet_in = self.handler.recv()
        return packet_in.split(",")
    
    def close_connection(self):
        '''End communication with client'''
        self.handler.close()
        return True
